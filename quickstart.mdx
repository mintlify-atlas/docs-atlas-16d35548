The fastest way to start a new `esp-hal` project is with `esp-generate`, the official project scaffold tool. It generates a ready-to-build project with the correct toolchain configuration, linker scripts, and dependencies for your chosen chip.

## Prerequisites

Before you begin, complete the [Installation](/installation) guide to set up your Rust toolchain and `cargo-espflash`.

## Create your project

<Steps>

<Step title="Install esp-generate">

Install the `esp-generate` tool from crates.io:

```bash
cargo install esp-generate
```

</Step>

<Step title="Generate a new project">

Run `esp-generate` with the `--chip` flag set to your target device. This example targets the ESP32-C6, but you can substitute any supported chip name:

```bash
esp-generate --chip=esp32c6 my-project
```

The generator will ask a few questions about optional features (Embassy async runtime, logging backend, etc.). For a basic blinky you can accept the defaults.

The generated project includes:

- A configured `Cargo.toml` with the correct `esp-hal` dependency and chip feature
- A `.cargo/config.toml` with the right build target and linker flags
- A `rust-toolchain.toml` (for Xtensa chips)
- A minimal `src/main.rs` entry point

<Tip>
  To see all available chips and generator options, run:

  ```bash
  esp-generate --help
  ```
</Tip>

</Step>

<Step title="Understand the blinky example">

Open `src/main.rs` in the generated project. A minimal blinky looks like this:

```rust src/main.rs
#![no_std]
#![no_main]

use esp_hal::{
    clock::CpuClock,
    gpio::{Level, Output, OutputConfig},
    main,
    time::{Duration, Instant},
};

// A panic handler is required for no_std binaries.
// esp_backtrace provides a more detailed handler; this minimal version
// performs a software reset on panic.
#[panic_handler]
fn panic(_: &core::panic::PanicInfo) -> ! {
    esp_hal::system::software_reset()
}

#[main]
fn main() -> ! {
    // Configure the system and set the CPU clock to the maximum frequency
    // supported by this chip.
    let config = esp_hal::Config::default().with_cpu_clock(CpuClock::max());

    // Initialize the HAL. This configures clocks, disables the watchdog
    // timers, and returns a `Peripherals` struct that gives you ownership
    // of every peripheral on the chip.
    let peripherals = esp_hal::init(config);

    // Configure GPIO0 as a push-pull output and drive it high initially.
    let mut led = Output::new(peripherals.GPIO0, Level::High, OutputConfig::default());

    loop {
        // Toggle the LED output between high and low.
        led.toggle();

        // Busy-wait for 500 ms using the HAL's monotonic time API.
        let delay_start = Instant::now();
        while delay_start.elapsed() < Duration::from_millis(500) {}
    }
}
```

Here is what each part does:

| Part | Purpose |
| :--- | :------ |
| `#![no_std]` | Disables the Rust standard library. Only `core` is available. |
| `#![no_main]` | Removes the standard `main` entry point. The `#[main]` macro from `esp-hal` defines the real entry point. |
| `#[panic_handler]` | Required by `no_std`. Called when the program panics. `software_reset()` reboots the chip. |
| `esp_hal::init(config)` | Initializes clocks, disables watchdog timers, and returns peripheral singletons. **Call this exactly once**, at the top of `main`. |
| `CpuClock::max()` | Sets the CPU to its highest supported frequency for the selected chip. |
| `Output::new(...)` | Configures a GPIO pin as a digital output. `peripherals.GPIO0` is the peripheral singleton for pin 0. |
| `led.toggle()` | Flips the output state between high and low. |
| `Instant::now()` and `Duration` | Monotonic time primitives from `esp_hal::time`. Used here to implement a non-blocking 500 ms delay without requiring a timer peripheral. |

<Note>
  Each peripheral singleton (like `peripherals.GPIO0`) can only be passed to one driver at a time. If you need to share a peripheral across multiple parts of your code, use the `reborrow()` method to create a shorter-lived handle without giving up ownership.
</Note>

</Step>

<Step title="Build the firmware">

Compile the project in release mode. Debug builds are significantly slower and can cause timing issues with hardware peripherals:

```bash
cargo build --release
```

The compiled ELF binary will be at `target/<your-target>/release/my-project`.

<Warning>
  Always use `--release` when building firmware for real hardware. The `dev`
  profile can be one or more orders of magnitude slower and may cause timing
  failures with peripherals.
</Warning>

</Step>

<Step title="Flash and monitor">

Connect your ESP32 board via USB, then flash and open the serial monitor in a single command:

```bash
cargo espflash flash --release --monitor
```

`cargo-espflash` will auto-detect the connected device's serial port, flash the firmware, and open a serial monitor. Press `Ctrl+R` to reset the device and `Ctrl+C` to exit the monitor.

If you have multiple devices connected, specify the port explicitly:

```bash
cargo espflash flash --release --monitor --port /dev/ttyUSB0
```

</Step>

</Steps>

## Explore more examples

The `esp-hal` repository contains a large collection of examples covering every supported peripheral. Browse them at [github.com/esp-rs/esp-hal/tree/main/examples](https://github.com/esp-rs/esp-hal/tree/main/examples).

<Tip>
  The repository uses `cargo xtask` to manage building, running, and testing
  examples across all supported chips. From the repository root, run the
  following to see available commands:

  ```bash
  cargo xtask help
  ```

  For example, to run the `blinky` example on an ESP32-C6:

  ```bash
  cargo xtask run --chip esp32c6 blinky
  ```

  When browsing examples on GitHub, always use the tag for the `esp-hal`
  version you have installed (e.g., `esp-hal-v1.0.0`) rather than `main`, as
  APIs on `main` may have changed since the last release.
</Tip>
