`esp-hal` supports nine ESP32 chips across two CPU architectures. This page lists all supported chips, their Rust target triples, and the current state of each peripheral driver.

<Note>
  The peripheral support table reflects the state of the **main branch**, not any stable release. Driver maturity varies вЂ” check the legend below before depending on a peripheral.
</Note>

## Chip features in Cargo.toml

Enable exactly **one** chip feature in your `Cargo.toml`. Enabling more than one is a compile error.

```toml Cargo.toml
[dependencies]
esp-hal = { version = "1.0.0", features = ["esp32c6"] }
```

The available chip feature names are:

| Feature | Chip |
|---|---|
| `esp32` | ESP32 |
| `esp32c2` | ESP32-C2 |
| `esp32c3` | ESP32-C3 |
| `esp32c5` | ESP32-C5 |
| `esp32c6` | ESP32-C6 |
| `esp32c61` | ESP32-C61 |
| `esp32h2` | ESP32-H2 |
| `esp32s2` | ESP32-S2 |
| `esp32s3` | ESP32-S3 |

## Checking peripheral availability at compile time

Some peripherals are gated behind chip-specific `cfg` flags generated at build time. If you write code that should compile for multiple chips, use conditional compilation to check availability:

```rust
#[cfg(esp32c6)]
{
    // ESP32-C6-only code, e.g. LP I2C
}
```

For more granular capability checks (e.g. whether PSRAM is available on the current chip), the build system exposes metadata-derived `cfg` flags вЂ” see the `esp-metadata` crate documentation.
